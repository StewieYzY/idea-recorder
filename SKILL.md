# Idea Recorder Skill

将对话中有价值的想法，结构化地记录到项目 `idea/` 目录中。

---

## 工作流程

### Step 1：确定项目根目录

```bash
# 优先使用 git 根目录
git rev-parse --show-toplevel 2>/dev/null || pwd
```

将记录写入 `<项目根目录>/idea/` 下。若无 git 仓库，则使用当前工作目录。

### Step 2：确定今天的文件路径

```bash
DATE=$(date +%Y-%m-%d)
IDEA_DIR="<项目根目录>/idea"
FILE="$IDEA_DIR/$DATE.md"
mkdir -p "$IDEA_DIR"
```

### Step 3：从对话中提炼内容

回顾本次对话（或当前话题的相关轮次），提炼以下六个维度的信息：

| 维度 | 说明 |
|------|------|
| **标题** | 用一句话概括这个想法的核心，简洁有力 |
| **确定性结论** | 人与 AI 共同达成的、人已明确认可的结论（非草稿，非建议）|
| **精华思考与辩论** | 讨论过程中出现的关键观点、反驳、取舍理由（提炼核心，不是全文复制）|
| **需求背景与应用场景** | 为什么需要这个想法？它解决什么问题？在哪些场景下适用？|
| **具体解法** | 这个想法的实现方案、技术路径或执行方式 |
| **后续待办** | 下一步需要做什么？有哪些未决问题需要跟进？|

**提炼原则：**
- 结论要明确，避免"可能""也许"等模糊表述
- 精华思考要保留"为什么这样决策"的逻辑，而不只是"决定了什么"
- 待办要可执行，尽量写成动词短句

### Step 4：写入 Markdown 文件

**如果当天文件不存在**，创建文件并写入文件头 + 第一条 idea。

**如果当天文件已存在**，在文件末尾追加新的 idea 条目（追加前加分隔线）。

写入格式见下方模板。

### Step 5：确认反馈

写入完成后，向用户简短确认：
- 文件路径
- idea 标题
- 如有待办，提醒用户关注

## Markdown 写入模板

### 文件头（仅当天第一条 idea 时写入）

```markdown
# 💡 Ideas — YYYY-MM-DD

> 记录于 [项目名称或路径]

---
```

### 每条 Idea 的结构

```markdown
## [序号]. [想法标题]

**记录时间：** HH:MM

### ✅ 确定性结论

[人与 AI 共同确认的结论，清晰、明确、可复述]

### 🧠 精华思考与辩论

[核心观点、关键取舍、重要反驳的提炼总结。保留"为什么"的逻辑]

### 🎯 需求背景与应用场景

**背景：**
[为什么会产生这个想法？当前面临什么问题或机会？]

**应用场景：**
[这个想法在哪里、什么时候、对谁有用？]

**目标：**
[这个想法希望达到什么效果？]

### 🔧 具体解法

[实现方案、技术路径、执行方式。如有多个方案，说明最终选择及原因]

### 📋 后续待办

- [ ] [待办事项 1]
- [ ] [待办事项 2]
- [ ] [待办事项 3（如有未决问题，也列在此）]

---
```

## 特殊情况处理

### 对话内容不够完整时
若某个维度在对话中没有被讨论（如还没想到具体解法），用以下占位符：

```markdown
### 🔧 具体解法

> 待补充 — 当前对话尚未讨论具体实现方案
```

不要强行编造内容，宁可留白也不要填充无用信息。

### 一次对话包含多个独立想法时
每个独立想法分别记录为单独的条目（按序号递增），不要合并。

### 用户要求修改已记录的内容时
读取对应日期的文件，定位到对应 idea 条目，进行局部更新，不要重写整个文件。

### 项目根目录无法确定时
提示用户："我没有找到明确的项目根目录，请问要把 idea 记录到哪个目录下？" 等用户确认后再写入。

## 注意事项

- **只记录已确认的结论**，不要把 AI 的建议草稿当作结论写入
- **保持精炼**：每条 idea 的总字数控制在 300–800 字之间
- **中立叙述**：记录时用第三人称或无主语短句，避免"我觉得""AI 说"等表述
- **时间准确**：写入时获取系统当前时间，不要估算
